## Vision

### 10x Check
AgentOS becomes the control plane for AI agent infrastructure — not just a config IDE but the platform where agent capability catalogs are built, verified, and discovered. The Registry is the compounding moat: every capability verified and contributed makes the platform more valuable for all future users. A team lead with 20 agents in production comes to AgentOS to understand what their fleet can do, not just to configure one agent at a time.

## Implementation Alternatives

### APPROACH A: Hybrid Card + Graph (RECOMMENDED)
**Summary:** Structured agent card with mini capability graph in Tools section, Discovery Panel as right sidebar, list/graph toggle.
- Effort: L (phased S→L→L→XL)
- Risk: Medium — biggest risk is capability DB being empty at launch
- Pros: Makes capability emergence visible, scales to 20+ tools, Discovery Panel is central, feels "alive" as you configure
- Cons: Content dependency (DB must exist before graph is useful), large surface area for v1
- Reuses: Wireframe at `/tmp/gstack-sketch-1743085568.html`

### APPROACH B: Flat Card First, Graph Later
**Summary:** Ship Config Studio with flat tool list + Discovery Panel. Capability graph comes after real user evidence.
- Effort: M (v1), L (add-on)
- Risk: Low — validates Discovery Panel before investing in graph complexity
- Pros: Faster to ship, lower initial scope, validates before committing
- Cons: Graph is the core differentiator — shipping without it may not be compelling enough

### APPROACH C: Discovery Panel as Primary UI
**Summary:** Flip the hierarchy. Agent list + Discovery Panel is the main view. Card is a drill-down from discovery.
- Effort: M
- Risk: High — inverts builder workflow
- Pros: Discovery-first mental model
- Cons: Most builders want to configure, not browse

**RECOMMENDATION:** Choose APPROACH A — the graph is load-bearing for the platform thesis. Without visible capability edges, the Discovery Panel is just a list and the Registry flywheel has no foundation.

## Accepted Scope

**v1 (this plan):**
- Hybrid Card UI: Memory/Config (left) + Tool Canvas with capability graph (center) + Discovery Panel (right)
- Capability graph: auto-draw edges when known capability's all toolIds are selected
- Tool picker: modal, searchable, grouped by category, one-action confirm
- Capability DB: 30+ seeded entries minimum at launch (founding team curation)
- Claude Code YAML export only
- Auto-layout: horizontal by category (Web→Files→Code→Memory→System), no force-directed
- Health check: HTTP HEAD, 2s timeout, non-blocking, per-tool dot (green/orange/red)
- No animation in v1
- Read-only graph visualization

**v1.1 (pending revenue model decision):**
- Team workspaces with cloud sync
- MCP export
- OpenAI JSON export
- History/audit trails (basic)

**v1.2:**
- Capability Registry (public, community-contributed)
- AI Generator-Evaluator pipeline
- Capability Verification harness

**v2:**
- Cost attribution per agent/team/project
- Compliance attestation (SOC2-ready exports)
- Enterprise RBAC, audit logs, immutable trails

## Tech Stack (pinned)

- **Framework:** React 18 + TypeScript
- **Bundler:** Vite
- **State:** Zustand (uiStore, agentConfigStore)
- **Storage:** IndexedDB via Dexie.js
- **Graph rendering:** Hand-rolled SVG (no force-directed library, auto-layout only)
- **Export:** js-yaml for YAML generation
- **Testing:** Vitest + Playwright (E2E)
- **v1:** Static CDN deploy, no backend, no auth, local-first

## Critical Gaps

1. **Capability DB empty at launch** — Graph is load-bearing. If 30+ entries don't exist at launch, the graph shows nothing. This is the single biggest delivery risk. Mitigation: flag as blocking dependency; if not ready, ship without graph.
2. **Analytics events not specced** — The kill condition (20% Discovery Panel engagement 90d post-v1.1) cannot be measured without instrumentation. Added via cherry-pick #3.
