## Vision

### 10x Check
AgentOS becomes the persistent control plane for AI agent infrastructure — not a config tool but the environment where agents actually run. The IDE metaphor earns its keep when the alternative is tmux grids and scroll-blind terminal output. Every agent spawned in AgentOS leaves a trace; every handoff is visible; every failure is caught by the Orchestrator before it cascades. The compounding moat is runtime lock-in: once agents run in AgentOS, their workflows, memory, and policies live there.

## Six Forcing Questions

### Q1: Demand Reality

**What it says:** "Karpathy described managing agent fleets with tmux grids. Every competitor has 1-2 pillars. No tool combines all three." Status: Early/Analogical.

**The CEO challenge:** "Karpathy" is a single anecdote. "Every competitor" is a category argument, not proof. "The gap is visible in how power users cope" — name one specific power user who told you they're coping this way. Analogical demand is the weakest form. What's the specific behavior that would prove this demand is real?

**What would change this:** If a user says "I literally have 4 tmux panes open right now and I hate it" — that's demand. If the exit criterion for v0.1 is "5 indie devs say they'd switch," that's a real behavioral test, not a survey. Is that the bar?

**Assessment:** The demand case is plausible but unvalidated. The 5-indie-dev exit criterion is the right test. The risk is that 5 devs saying "I'd switch" and 5 devs actually switching after 2 weeks of rough edges are different things.

---

### Q2: Status Quo

**What it says:** "tmux grids + custom scripts." The workaround is fragile, manual, non-visual, isolated.

**The CEO challenge:** Is "I use tmux" actually painful enough to switch? Most power users who use tmux actually *like* tmux. The pain isn't the tmux grid — it's when something fails and you don't see it. Does the status quo have a name and a face, or is it a generic category?

**Assessment:** The tmux grid pain is specific and real for multi-agent users. This is the strongest part of the demand case because it's describing an actual workflow, not a hypothetical.

---

### Q3: Desperate Specificity

**What it says:** "Indie dev / solo builder (3-5 personal agents)."

**The CEO challenge:** Not a name. Not a company. "Indie dev" is a segment, not a person. What's the actual person who needs this today? What are they building? What would make them pay $20/month vs. using tmux for free?

**The narrowest wedge framing is right.** "See all my agents on a canvas with live state" is a concrete value prop. But who is the first 10 users? YC-backed founders with 3+ agents? AI hobbyists building side projects? Something else?

**Assessment:** The wedge is correctly narrowed (indie dev, not enterprise). But the specific persona — what they're building, what their agent workflows look like, what they tried before — needs to be sharper before fundraising or scaling.

---

### Q4: Narrowest Wedge

**What it says:** "Visual agent canvas — see all agents on a canvas with live state, connect them, see capability edges emerge."

**The CEO challenge:** Is THIS the thing worth $0? Worth $20/month? Or is the narrower wedge just "I want to run my Claude Code session in a tab and see its state"? The canvas with handoff edges is the most visually impressive feature — and the hardest to explain in a single sentence. If someone asks "what does it do?" and the answer is "it's like VS Code for agents," that's a $100M positioning budget away from clarity.

**Try this instead:** "It runs your AI agents and shows you what they're thinking — every tool call, every handoff, every failure — in one window." Not "canvas." Not "IDE."

**Assessment:** The canvas is the right differentiator, but the value articulation for a non-technical buyer ("it shows agents on a canvas") doesn't survive first contact. The Orchestrator is actually the more compelling wedge for team leads (watch all my agents) — but that's Phase 3, not Phase 1.

---

### Q5: Observation & Surprise

**What it says:** No user research conducted yet.

**The CEO challenge:** Have you watched anyone use a tmux grid for agent management? Have you seen someone try to explain their multi-agent setup to a colleague? What surprised you about what agents actually do when they're running?

**Assessment:** No observation has been done yet. The plan correctly identifies this as "analogies" not evidence. The first real evidence is the 5-dev exit criterion in v0.1.

---

### Q6: Future-Fit

**What it says:** "AI agents are autonomous processes." As agent frameworks proliferate (Claude Code, Cursor, Copilot Workspace), the need for agent infrastructure grows.

**The CEO challenge:** "AI agents need an IDE" assumes agents stay in development/tooling context. But what if the future is agents running in production infrastructure (agents as services, not desktop tools)? Then AgentOS as a desktop IDE becomes a development-time tool, not a runtime control plane. Is the desktop the right anchor, or should this be server-side?

**Assessment:** The desktop-as-anchor is both a strength (no cloud dependency, full process control) and a risk (if agents move to server-side infrastructure, the TAM shrinks). The plan correctly identifies that Claude Code-style terminal-first has won in the market — but that product is solo and session-based. The gap AgentOS fills (multi-agent, visual, persistent) is real — but it's a bet that multi-agent setups become common enough to need orchestration.

## Glossary

| Term | Definition |
|------|------------|
| **Capability edge** | A labeled connection between two agents on the canvas representing a known handoff or data flow. Example: "Research Pipeline" = researcher agent → verification agent. Displayed as a labeled directed edge. |
| **Live state** | Real-time status of an agent captured via IPC events (spawn, stdout data, tool call, error, exit) flowing from the agent process to the renderer via MessagePort. Not a polling mechanism — push-based via Electron IPC. |
| **Watch-and-learn** | Orchestrator mode in Phase 3 where it observes agent event patterns (tool call durations, error rates, handoff delays) for 24-72 hours before surfacing policy suggestions. Evolves into predictive mode as patterns accumulate. The Orchestrator dashboard shows observed patterns with confidence scores. Users accept or dismiss suggestions. Accepted policies are stored in SQLite and evaluated against live agent events. |

## Vision Assessment

**Is this the right thing to build?** Yes — the gap is real and visible. The tmux grid workaround is the clearest demand signal. The multi-agent canvas is the right core positioning.

**Is this the right time?** Yes — Claude Code has validated the "agent as terminal-native process" mental model. The market is warm. The tooling (Electron, React Flow, xterm.js, MCP) is mature. No one has combined all three pillars yet.

**Is this the right person to build it?** Partially — the solo engineer has a massive scope (Electron + React + MCP + PTY + SQLite + canvas). The Phase 0 runtime sprint is the right first proof-of-concept. If it takes longer than 2 weeks, the scope risk is real.

## What Survived the Review

1. **The three pillars are the right frame.** Canvas + Runtime + Orchestrator is not incremental — it's a new category claim. The claim is credible because no one else has all three.

2. **"Watch and learn" solves the right problem.** Beginners don't know what policies to configure. The solution (observe first, suggest second) is correct. The risk is that "watching" for 24-72 hours before getting value is too slow for users who need help now.

3. **Electron is the right technical anchor.** Desktop-first avoids cloud complexity, enables full process control, and the dev tools audience accepts installed apps. The risk is Mac App Store distribution (notarization, signing).

4. **The kill condition is right in spirit.** If indie devs don't place ≥3 agents on the canvas in 30 minutes, the UX is wrong. But it should measure task completion, not just canvas interaction.

## Competitive Landscape

**Cling Kanban** (announced 2026-03) is the most direct competitor — a browser-based Kanban board for orchestrating coding agents using git worktrees. Validates the core pain: "20 terminal windows open." Target: developers running Cline/Claude Code/Codex in parallel. Limitation: coding-only, session-based, git worktree isolation.

**Implications for AgentOS:**
- The demand is real — multiple products independently converging on the same problem
- AgentOS's wedge vs Cling: real-world business agents (sales, ops) vs coding agents
- Not browser-based: native Electron gives full process control, not just terminal tabs
- MCP-native: connects to business-relevant servers (filesystem, github, memory, not just code)
- Persistent agents: memory and state survive sessions, unlike Cling's ephemeral worktrees
- Canvas + Orchestrator + Runtime: all three pillars, where Cling has only the orchestration layer

**Assessment:** Cling validates. AgentOS is differentiated by scope (business agents, not coding), persistence (memory survives sessions), and breadth (all three pillars). The coding-agent niche is contested — AgentOS should own the business-agent niche.
