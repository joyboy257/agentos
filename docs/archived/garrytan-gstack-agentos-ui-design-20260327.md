## Problem Statement

IDEs are the wrong metaphor for AI agents. Two problems:

1. The wrong user: IDEs serve developers building code. The team lead managing a fleet needs a dashboard or list view, not a code editor.
2. Agents aren't code: Agents are dynamic, stateful, probabilistic. An IDE implies writing instructions. The right mental model is not "write code for an agent" — it is "equip an agent with tools and watch it become capable."

The hybrid UI solves this by making capability emergence **visible** rather than **abstract**. Builders see what their tool selection enables, not what the YAML schema says.

## Demand Evidence

**Status: Unvalidated** — this is the core work of the next phase.

- Current signal: No explicit builder demand for visual capability graphs yet confirmed
- Analogous validation: n8n's node canvas is widely used; Linear's clean list views win over complex project management tools; the pattern of "visual composition + structured detail" is proven in adjacent categories
- Next validation: Show the wireframe to 5 builders and ask if they'd switch their workflow

## Target User & Narrowest Wedge

**Who:** Builders configuring complex AI agents — solo indie devs with 3-5 multi-tool agents, and team leads building agent templates for their team.

**The narrowest version:** Config Studio + Discovery Panel + capability graph, scoped to single-agent configuration in v1.

**What "complex" means:** Agents with 3+ tools where capability emergence is non-obvious. Simple 1-2 tool agents don't need a graph — they're a checklist. The graph earns its pixels at 3+ tools.

**Critical nuance:** The graph serves solo devs building single agents in v1. Fleet-level capability overview (seeing all team agents' capability compositions at once) is a v2 feature requiring team workspaces. These are separate UIs, not one toggle.

## Constraints

- Must make the Discovery Panel's value proposition **visible**, not abstract
- Must scale to 20+ tools without becoming unreadable (threshold found empirically via v1 instrumentation)
- Must not slow down the core workflow — builders configure agents, they don't browse them
- The graph is **read-only visualization** of capability composition, not a full canvas editor
- Tools are added via picker, not drag-to-canvas
- No animation in v1 — edges appear/disappear immediately
- Fleet capability overview is explicitly out of scope for v1

## Premises

1. **IDEs are wrong for AI agents** — confirmed with caveat: a well-designed IDE-like UI is fine, but code-editor metaphors (YAML, file trees, debug panels) don't fit
2. **Visible capability edges beat abstract lists** — the graph makes the Discovery Panel tangible, not just a sidebar
3. **Graph is v1** — it is the core differentiator; without visible capability edges, Discovery Panel is just a list
4. **The hybrid is the right balance** — between n8n's full canvas (too complex) and flat card (not visible enough)

## Approaches Considered

### Approach A: Hybrid Card + Graph (Recommended)

Structured agent card with mini capability graph in Tools section, Discovery Panel as right sidebar, list/graph toggle.

- Effort: L | Risk: Medium
- Pros: Makes capability emergence visible, scales reasonably, Discovery Panel is central
- Cons: Graph + Discovery Panel + capability DB is a large surface area for v1; content dependency (DB must exist before graph is useful) is the biggest risk

### Approach B: Flat Card + Discovery Panel First, Graph as v1.1

Ship Config Studio with flat tool list + Discovery Panel. Capability graph comes after real user evidence.

- Effort: M (v1) | Risk: Low
- Pros: Faster to ship, validates Discovery Panel before investing in graph complexity
- Cons: Graph is the core differentiator — shipping without it may not be compelling enough
- Note: If graph is truly load-bearing (Premise 3), Approach B is not a valid alternative — it's Approach A without the main feature. Validate the graph's load-bearing status before choosing.

### Approach C: Discovery Panel as Primary UI, Card as Detail

Flip the hierarchy. Agent list + Discovery Panel is the main view. The card is a drill-down from discovery.

- Effort: M | Risk: High
- Pros: Discovery-first mental model — browsing capabilities, not filling a form
- Cons: Inverts the builder workflow — most builders want to configure, not browse

## Recommended Approach

**Approach A: Hybrid Card + Graph**

The wireframe (saved at `/tmp/gstack-sketch-1743085568.html`) shows the layout:

```
┌──────────────────────────────────────────────────────────────────────┐
│  Agent: Research Agent                            [Export ▾] [Save]  │
│  ──────────────────────────────────────────────────────────────────  │
│                                                                       │
│  Memory & Context      Tools                      Reliability         │
│  ─────────────────    ─────                      ──────────         │
│  ● Persistent session  ┌────────────────┐          Max steps: 50    │
│    Context: 200k tokens│  [Web Search]──┼──→Synthesis             │
│    History: last 50    │       │         │           Retry: 3×    │
│                        │  [File System]─┼─→Code Runner             │
│  Tasks                 │       │         │           Timeout: 30min│
│  ─────                 │  [Browser]───────→Research Pipeline       │
│  ● Long-running enabled└────────────────┘           ● Health: OK    │
│  ● Checkpoint: every 50 steps                                         │
│                                                                       │
├──────────────────────────────────────────────────────────────────────┤
│  DISCOVERY PANEL                                                     │
│  Ready: ● Research Pipeline  ● Synthesis                            │
│  Partial: ○ Synthesis → needs: Code Interpreter                      │
│  Unverified: ◌ Multi-hop Reasoning (AI-generated)                     │
└──────────────────────────────────────────────────────────────────────┘
```

**How it works:**

- Tools section = mini canvas showing tool nodes + capability edges
- Edges draw automatically when 2+ tools that compose a known capability are both selected
- Click edge → highlights Discovery Panel card (bidirectional)
- Click Discovery card → highlights the tools that compose it
- List/Graph toggle lets users collapse graph for simple agents
- Tools added via picker, not drag-to-canvas
- Graph is read-only — capability editing happens in Discovery Panel

**What "Reliable" looks like in the UI:**

| Config | Meaning |
|--------|---------|
| Max steps: 50 | Agent won't spiral indefinitely |
| Timeout: 30min | Agent won't hang forever |
| Retry: 3× | Agent recovers from transient failures |
| Checkpoint: every 50 steps | Long tasks resume, not restart |
| ● Health: OK | Tool's API endpoint is reachable at config time (HTTP HEAD, 2s timeout, non-blocking) |

**What "live" means:** Edges re-draw immediately when a tool is added or removed via the picker. Health dots refresh on tool selection. No animation on edge appear/remove in v1 — deferred to v1.1 if user research demands it.

**What "capability" means:** A named composition of 1+ tools that enables a specific agent function. Capabilities are defined in the capability DB (seeded by Generator-Evaluator pipeline in v1.2, curated static entries in v1). The graph queries: given selected toolIds, find all capability entries where toolIds ⊆ selectedTools.

**How edge detection works:** The capability DB contains entries like:

```typescript
interface Capability {
  id: string;
  name: string;                    // "Research Pipeline"
  description: string;             // "Cross-reference claims across sources..."
  toolIds: string[];               // ["web_search", "file_system", "browser"]
  emergent: boolean;              // requires 2+ tools
  coverage: "core" | "extended" | "experimental";
  provenance: "curated" | "ai_generated";
  status: "verified" | "unverified";  // human-reviewed or not
}
```

Edge drawing: for each capability where ALL toolIds are selected, draw edges connecting each selected tool to a visible capability node. Tools do not connect directly to each other — they connect to the capability label. The capability label appears at the midpoint of the edges.

**Capability state machine:**

| State | Condition | Card appearance | Edge |
|-------|-----------|-----------------|------|
| **Ready** | All toolIds selected AND verified | Green left border | Solid edges to each tool |
| **Partial** | Some toolIds selected | Orange left border | Dashed edges to selected tools; "Needs: X" shown |
| **Unverified** | Any state, provenance = ai_generated | Gray left border + "AI-generated" badge | Per state above |

A capability can be Ready + Unverified (all tools selected, AI-generated, not yet human-reviewed). The statuses are orthogonal: provenance (curated/AI) vs. completeness (Ready/Partial).

**Tool picker:** Modal dialog triggered by "+ Add Tool" button in the Tools section header. Searchable list of available tools grouped by category (Web, Files, Code, Memory, etc.). Click to select, click again to deselect. Confirm adds all selected tools to the agent config and closes the modal. No multi-step wizard — one action to add.

**Auto-layout in v1:** Tools arranged in a horizontal row by category order (Web → Files → Code → Memory → System). Capability labels appear above the row, connected by thin lines to the tools that compose them. No force-directed simulation — this is a deterministic single-pass layout. Simplicity is the point.

**Edge Cases & Error States:**

| Situation | Behavior |
|-----------|----------|
| Capability loses a required tool (user removes one) | Edge disappears immediately; Discovery Panel updates to Partial or removes the card |
| Tool health check fails | Orange dot on tool node; "Health: unreachable" label; does not block save |
| 20+ tools selected | Graph degrades gracefully: edges remain but auto-layout may overlap; Discovery Panel filter becomes primary interface |
| Zero capabilities detected | Graph shows tool nodes only; Discovery Panel shows empty state with hint text |
| Graph render fails | Fallback to list view silently |

**v1 scope for the graph:**

- Tool nodes: small circles with icon + name
- Capability edges: auto-draw when known capability detected (pure lookup, not ML)
- Click edge → highlight Discovery card
- Click Discovery card → highlight relevant tools
- Auto-layout: layered layout (tools arranged in rows by category), no drag-to-rearrange in v1
- No drag-to-add tools — tools added via picker only
- List/Graph toggle (defaults to Graph — "alive" feeling is the point)
- Discovery Panel is always-on right sidebar in v1 — not a browse/drill-down mode

## ⚠️ Blocking Dependency: Capability DB Must Launch With v1

The graph is read-only visualization of pre-existing capability data. If the capability DB is empty or sparse at launch, the graph shows nothing and the Discovery Panel has no cards.

**Minimum viable DB for launch:** 30+ capability entries covering the 12 most common tools (Web Search, File System, Browser, Code Interpreter, Memory, etc.). This is a content curation problem, not an engineering problem.

**Who owns this:** The founding team seeds the DB before launch. The Generator-Evaluator pipeline (v1.2) enables community contributions, but the initial corpus is hand-curated.

**If the DB is not ready:** Ship v1 without the graph. The Discovery Panel still works as a list. Re-add the graph once the DB has substance.

This is the single biggest delivery risk for the UI described in this doc.

## Open Questions

1. **Graph at 20+ tools:** Beyond this threshold the mini canvas becomes unreadable. Discovery Panel filter/search becomes the primary interface. Find the exact threshold via instrumentation in v1.

2. **Fleet capability overview:** Explicitly deferred to v2. The v1 graph is single-agent only.

3. **Graph vs. list default:** **Decision: Graph is default-on.** List-only available via toggle.

4. **Animation:** **Decision: No animation in v1.** Edges appear/disappear immediately.

5. **Memory as graph edge:** **Decision: Structured config row only.**

## Success Criteria

- 5 builder interviews with wireframe: 4/5 say they'd use this to configure agents
- Graph readability threshold found: at what tool count does the canvas become unreadable?
- List/Graph toggle: does it get used, or do users settle on one mode?
- Discovery Panel engagement: % of sessions where user clicks through to a capability card
