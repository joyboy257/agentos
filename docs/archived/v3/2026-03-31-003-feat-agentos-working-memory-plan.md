# Working Memory for Agent Session Context

## Overview

Implement a session-scoped working memory system so agents can remember prior interactions within a run. This gives agents context without requiring long-term persistence — useful for multi-turn conversations, follow-up questions, and maintaining conversation state.

## Problem Frame

Currently AgentOS agents are stateless within a run. If a user sends 5 messages to a "customer support" agent, each message starts fresh — the agent doesn't remember earlier context from the same session.

**Use cases:**
- User asks "show me recent leads" then follows up "which of these are in healthcare?" — agent needs context from first query
- Multi-step research: agent reads emails, then drafts response using context from what it read
- Canvas has a chat panel where user and agent exchange multiple messages — agent must track conversation history

## Requirements Trace

- R1 (from MVP): NL-to-canvas pipeline builder — working memory enables multi-turn agent conversations on canvas
- R6 (from MVP): Reasoning trace per run — working memory entries are visible in the trace as `memory` events

## Key Technical Decisions

**Decision: In-memory with optional Redis backend.**
- Rationale: MVP uses in-memory. Production uses Redis for persistence across server restarts and multi-instance deployment.
- Future: Add Redis backend post-MVP without changing the interface.

**Decision: Mark-based filtering as primary retrieval pattern.**
- Rationale: Simple, typed, no query language needed. Marks like `user`, `agent`, `system`, `hidden` cover most filtering needs.
- Alternative: Full-text search — rejected for v1 (adds complexity without clear MVP need).

**Decision: Memory is append-only within a session.**
- Rationale: Simpler implementation, no tombstone management. Clear memory by ending session or explicit `clear()`.

## Scope Boundaries

- **Long-term memory** — out of scope (separate microservice, see LT Memory plan)
- **Memory persistence across server restarts** — out of scope for MVP (in-memory only)
- **Memory encryption** — out of scope (PII already handled at the PII redaction layer)
- **Token count enforcement** — out of scope (hard limit on entries, not tokens)

## High-Level Technical Design

```
┌─────────────────────────────────────────────────────────────────┐
│                      Agent (per-run)                              │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │                   WorkingMemory                              │ │
│  │  entries: MemoryEntry[]                                    │ │
│  │  marks: Map<mark, entryId[]>                               │ │
│  │                                                            │ │
│  │  add(entry, marks?)                                        │ │
│  │  get(mark?, excludeMark?): MemoryEntry[]                  │ │
│  │  delete(ids): count                                        │ │
│  │  clear()                                                   │ │
│  │  getRecent(count): MemoryEntry[]                            │ │
│  └────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────┘
```

### MemoryEntry Schema

```typescript
interface MemoryEntry {
  id: string              // ULID
  agentId: string
  role: 'user' | 'agent' | 'system'
  content: string        // Text content
  marks: string[]         // e.g. ['user', 'visible']
  toolName?: string      // If generated by a tool call
  createdAt: number      // Unix timestamp
}

interface MemoryStore {
  add(entry: MemoryEntry, marks?: string[]): Promise<void>
  get(filter?: { mark?: string; excludeMark?: string }): Promise<MemoryEntry[]>
  getRecent(count: number): Promise<MemoryEntry[]>
  delete(ids: string[]): Promise<number>
  clear(): Promise<void>
}
```

### Mark Conventions

| Mark | Meaning |
|------|---------|
| `user` | Message from the user |
| `agent` | Response from an agent |
| `system` | System message (e.g., "run started") |
| `hidden` | Not shown in canvas chat UI |
| `reasoning` | Agent internal reasoning (not shown to user) |
| `approved` | Tool call was approved by human |
| `rejected` | Tool call was rejected by human |

## Implementation Units

- [ ] **Unit 1: WorkingMemory Core**

**Goal:** In-memory working memory store with mark-based filtering.

**Requirements:** R1

**Files:**
- Create: `lib/memory/working-memory.ts` (WorkingMemory class)
- Create: `lib/memory/types.ts` (MemoryEntry, MemoryStore, MemoryMark)
- Create: `lib/memory/index.ts`
- Test: `lib/memory/__tests__/working-memory.test.ts`

**Approach:**
- `WorkingMemory` class implements `MemoryStore`
- Entries stored in `Map<string, MemoryEntry>` (by ULID)
- Marks stored in `Map<string, Set<string>>` (mark → entry IDs)
- `add()` pushes to marks index
- `get(mark?)` returns filtered slice, preserving insertion order
- Thread-safe via mutex for production readiness

**Patterns to follow:**
- AgentScope `agentscope/src/agentscope/memory/_working_memory/memory.py` for API design
- `lib/tracing/event-buffer.ts` for ULID-based entry management

**Test scenarios:**
- `add()` then `get()` returns the entry
- `get(mark='user')` returns only user entries
- `get(excludeMark='hidden')` excludes hidden entries
- `delete(ids)` removes entries and cleans up marks
- `clear()` empties all entries

**Verification:**
- `npx vitest --run lib/memory/__tests__/working-memory.test.ts` passes

## System-Wide Impact

- **Runner:** New `workingMemory` dependency injected into agents
- **Tracing:** Memory events flow through SSE alongside trace events
- **Canvas:** Reasoning panel gains memory entry rendering
- **A2A (future):** A2A agents could receive memory context in their task payload

## Risks & Dependencies

- **Memory growth:** Unbounded memory entries could exhaust server RAM. Mitigation: `MAX_ENTRIES` limit (default 1000) with oldest-entry eviction.
- **PII in memory:** User messages in memory could contain PII. Mitigation: PII redaction layer processes entries before storage.
- **Redis in MVP:** MVP uses in-memory. If production uses Redis, session data changes behavior (persistent vs. in-memory). Mitigation: same interface, documented behavioral difference.
