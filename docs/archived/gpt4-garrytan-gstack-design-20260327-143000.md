## Problem Statement

Developers managing multiple AI agents lose track of what each one is doing, what tools they have access to, and what state they're in. The current workaround is multiple terminal windows and mental tracking — the same chaos that drove developers to bug trackers before project management tools existed. People are reaching for Linear as a proxy, but Linear is designed for human task management, not AI agent orchestration. The mental model is wrong.

## Demand Evidence

Personal pain, first principles. User feels the friction daily: switching between terminals, no shared state, agents drifting out of sync. Hours lost per week is unquantified but constant. The friction is worst when context-switching between multiple agent sessions — the cognitive load of remembering which agent is where, what it's working on, and what it's capable of.

## Status Quo

Multiple terminal windows. Each agent runs in its own terminal with its own CLI session. State is ephemeral — logs scroll by and disappear. No shared memory between agents. Tool access is configured per-agent via separate config files or command-line flags. Discoverability of agent capabilities is documentation-driven: you read the docs, you guess, you try, you iterate.

## Target User & Narrowest Wedge

**Who:** Developers building with AI agents — from indie hackers running 2-3 agents to teams managing complex agent pipelines.

**Narrowest wedge:** An Agent Creation Studio — a GUI for building/configuring agents, assigning roles, and defining capabilities. The first thing worth paying for is a better way to BUILD agents (not just observe them running). The GUI surfaces capabilities you didn't know existed — discoverability as the core value, not just prettier config.

## Constraints

- Must feel design-first, not dev-tool-ugly — aim for Linear quality, but if solo dev without a designer: ship with shadcn/ui defaults and iterate on user feedback rather than die on polish
- Must be agent-native: models how agents actually work (tool-using, stateful, probabilistic)
- Must integrate with existing agent frameworks (can't reinvent the agent runtime)
- Discovery must be genuinely useful — not just a feature, but the core differentiator

## Premises

1. **Other developers share this pain.** Terminal-wielding chaos is representative, not idiosyncratic. As agent proliferation increases, this problem gets worse, not better.
2. **Agent creation, not orchestration, is the right wedge.** The first thing people pay for is a better way to BUILD agents. Observability is secondary.
3. **Discoverability is the durable moat.** A GUI that reveals hidden emergent capabilities (from tool combinations) is defensible. Competitors can copy a YAML editor; they can't easily copy a discovery engine that understands how tools compose.

## Approaches Considered

### Approach A: Minimal Viable — Single Agent Config Studio

A focused web app for creating and managing one agent at a time. Define role, capabilities, system prompt, and tools via a form-based GUI. Export to standard YAML/JSON.

```
Effort: M (1 dev, 4-6 weeks) | Risk: Low
Pros: Ship fast, validate the creation thesis quickly, works with any agent framework
Cons: Doesn't solve the "multiple agents, losing track" problem directly
Reuses: Standard YAML/JSON config formats
```

### Approach B: Full IDE — Multi-Agent Orchestration Canvas

Full visual IDE with a canvas for designing agent workflows. Agent nodes, connection lines, shared memory, real-time state panel. n8n-style + VS Code.

```
Effort: XL (3+ devs, 6+ months) | Risk: High
Pros: Complete solution, hardest to replicate
Cons: 6+ months to build anything impressive, complex to maintain
Reuses: n8n workflow patterns, VS Code extension model
```

### Approach C: Discovery-First Agent Explorer

Flip the premise: instead of "build agents," start with "discover what agents can do." An interactive browser that connects to LLMs and tools and surfaces capabilities from combinations. Agents built by composing discovered capabilities.

```
Effort: M (1 dev, 4-6 weeks) | Risk: Med
Pros: Novel, directly delivers on discoverability thesis, defensible
Cons: Less obvious product story, harder to explain
Reuses: LLM API integrations, static capability database (curated)
```

Note: Effort estimates assume 1 fullstack developer. M = solo buildable, XL = requires team.

**v1 Scope (Hybrid A+C):** Ships with Single Agent Config Studio + Discovery Panel backed by static curated DB. LLM-in-the-loop discovery is v1.1. Design target: 6-8 weeks for v1, including 1-2 weeks pre-build research to populate the capability DB.

## Recommended Approach

**Hybrid A+C** — Agent Creation Studio (A) with Discovery built in from day one (C). Start with single-agent creation: role, system prompt, tool assignment. But the Discovery panel is always visible, surfacing emergent capabilities from tool combinations as you configure. This delivers immediate value (better agent config) while building the discovery moat incrementally.

The visual direction is **design-first dark UI** — Linear/VS Code-inspired aesthetics. Not a dev tool that looks like a terminal wrapper; a product that feels designed. Specific references: VS Code's sidebar + panel layout, Linear's sidebar navigation and modal patterns, Raycast's focus on keyboard shortcuts and density-without-cramping. Dark mode default (like VS Code), indigo accent color (#6366f1), monospace for code/tool output, system UI font for labels. Component library: shadcn/ui as starting point — avoids building from scratch while allowing Linear-quality customization.

## Data Model

```typescript
interface AgentConfig {
  id: string;
  name: string;
  role: string;                    // e.g., "Web Research Specialist"
  systemPrompt: string;
  tools: string[];                 // tool IDs assigned to this agent
  metadata: {
    createdAt: string;
    updatedAt: string;
    lastRun?: string;
  };
}

interface ToolDefinition {
  id: string;                      // e.g., "web-search"
  name: string;                    // e.g., "Web Search"
  description: string;
  category: "browser" | "api" | "filesystem" | "communication" | "data";
  schema: Record<string, unknown>;  // tool input schema
}

interface CapabilityMapping {
  id: string;
  name: string;                    // e.g., "Multi-source synthesis"
  description: string;
  toolIds: string[];               // tools that enable this capability
  emergent: boolean;               // true if capability needs ≥2 tools
}
```

## Discovery Flow

**Trigger:** When user selects or deselects a tool in the config panel, the Discovery engine:
1. Reads current `tools[]` from agent config
2. Queries the Capability DB for all `CapabilityMapping` where `toolIds` is a subset of selected tools
3. Renders capability cards in the Discovery panel — sorted by emergent-first (capabilities needing 2+ tools surface at top)
4. For capabilities where `emergent: true` and all required tools are selected: highlighted as "Ready to use" (green border)
5. For capabilities where required tools are partially selected: shown as "Needs: [missing tool]" — clickable to auto-add

**Note:** LLM-in-the-loop (Option B) is deferred to v1.1. v1 ships with static curated DB only. This keeps v1 scope manageable.

## Error Handling & Edge Cases

- **Tool conflicts:** Two tools modifying the same file path — not handled in v1, surface in v1.1 based on user feedback
- **Empty config:** Disable export button until agent has name + at least one tool selected
- **Capability DB miss:** If user selects an unknown tool, Discovery panel shows "No capabilities mapped yet — contribute one"
- **Large configs:** Agent config maxes at 50 tools; warn at 40+ tools about context window implications
- **LLM API failure (future v1.1):** Graceful degradation to static DB results; no error shown to user

## Open Questions

1. **Agent runtime:** Do we build the agent execution engine or only the config UI? **Decision: Config UI only** — integrate with existing runtimes. Export to YAML/JSON format usable by Claude Code (via `.claude/settings.json` agent definitions), OpenAI Assistants API, and any MCP-compatible runtime.

2. **Discovery mechanism:** How does the discovery panel work? **Option A (simpler):** Static capability database — curated mapping of tool→capabilities, updated periodically. **Option B (better but harder):** LLM-in-the-loop — as user selects tools, query the LLM to reason about emergent capabilities from combinations. Option A ships faster; Option B is the real moat. Recommendation: ship Option A, research Option B.

3. **Multiplayer:** Solo-first or team-first? **Decision: Solo first.** Teams require auth, permissions, conflict resolution. Ship solo, validate demand, then add multiplayer if teams ask for it.

4. **Distribution:** Web app vs. desktop Electron? **Decision: Web (local-first) first.** Electron adds build complexity. Ship as a local web app that runs entirely in the browser with IndexedDB for storage. Optional cloud sync (Excalidraw-style) comes later.

5. **Tool ecosystem:** MCP is emerging but many agents use OpenAI function calling or custom tool definitions. **Decision:** Support both. MCP as the primary integration target (future-proof), but also support YAML-based tool definitions that map to OpenAI function calling format.

## Success Criteria

- **Agent creation time:** Developer can create a new agent with role, system prompt, and 3+ tools in < 5 minutes (self-measured by completing the flow and timing)
- **Discovery utility:** 70%+ of users (from 10-user beta) report discovering at least 1 tool capability they didn't know their selected tools could do (survey after first use)
- **Config portability:** Exports to Claude Code YAML format and OpenAI Assistants JSON format — verified by manual integration tests against both platforms
- **Perceived quality:** 80%+ of beta users rate the design quality as "feels like a real product, not a dev tool" (5-point Likert scale, ≥4)
- **v1 success bar:** 100+ agents created by beta users within first 2 weeks of launch

## Dependencies

- **Design system:** shadcn/ui as component foundation. Achieving Linear-quality polish requires 4-6 weeks of design iteration if no dedicated designer — or 1-2 weeks if a designer is available. Without a designer, use the shadcn/ui defaults and iterate based on user feedback rather than trying to match Linear from day one.
- **Agent runtime integrations:** MCP (primary, future-proof), OpenAI function calling (legacy compatibility), Claude Code YAML format. Tool definitions stored as structured JSON.
- **Discovery capability database:** Initial curated database of 50-100 tool-to-capability mappings. Built manually by the founding team, not AI-generated (quality matters more than coverage initially). Can be community-contributed over time.
- **Local-first architecture:** IndexedDB for storage, no backend required for core functionality. Cloud sync is a future feature — do NOT build it in v1. This keeps it simple and keeps user data local.

## Reviewer Concerns

*Issues identified after 3 rounds of adversarial review — unresolved, flagged for implementation team*

1. **Capability DB population effort is likely undercosted.** 1-2 weeks to manually build 50-100 high-quality tool→capability mappings is optimistic. The curator must understand each tool deeply. Budget 3-4 weeks and treat it as a pre-launch blocker — the Discovery panel is the moat, and a thin DB weakens it.

2. **MCP integration deferred to v1.1 — may limit early user appeal.** If the target user is Claude Code power users, lack of MCP export in v1 means they can't use their existing MCP toolchains. Consider whether MCP export should be v1 despite the complexity.

3. **No security review of stored system prompts.** Agent configs stored in IndexedDB may contain sensitive instructions, API keys, or proprietary workflows. v1 should at minimum note that local storage is unencrypted and advise against storing secrets.
