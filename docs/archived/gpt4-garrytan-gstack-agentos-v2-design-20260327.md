## tl;dr

AgentOS is not an IDE for solo developers running a few personal agents. It is the **operating system layer** for businesses that deploy, manage, and orchestrate AI agents at scale. The buyer is the team lead, startup, or enterprise that has agents running in production and needs to manage them with the same rigor they'd apply to any critical infrastructure: visibility, control, compliance, cost attribution, and eventually a marketplace for agent capabilities.

The v1 "Agent Creation Studio" thesis is validated as the wedge — it's the entry point that proves the category. But the vision is larger: a platform where agents are built (Studio), verified (Capability Registry), deployed (MCP integration), monitored (runtime telemetry), and eventually transacted (Agent Marketplace). The "LinkedIn for agents" framing is a 2028+ concept. The immediate opportunity is being the **infrastructure layer** between agent builders and agent operators.

## Problem Statement

**The problem is not building agents. The problem is managing them.**

Every team we've talked to that deploys AI agents in production has the same story: they started with one agent, it worked, they added more. Now they have 10-40 agents running across the organization — different teams, different purposes, different frameworks — and they have **no visibility into what any of them are doing**. Agents are failing silently. API keys are scattered across configs. There are no audit trails. Compliance teams are asking questions nobody can answer. The agents exist. The infrastructure to run them properly doesn't.

This is exactly where software development was before Jira: teams shipped code without issue tracking, sprint planning, or visibility. Jira didn't make developers more productive — it made the *organization* able to manage what developers were producing. The same gap exists for agents.

**The secondary problem**: There's no universal framework for building agents. OpenClaw showed what's possible but is a "chimera mess" of hobbled-together features (per the user's direct assessment). Claude Code and Codex are session-based and solo-focused. CrewAI, AutoGen, LangFlow, and Dify are orchestration frameworks, not management platforms. The "agent IDE" category doesn't exist yet.

## Status Quo

**What teams are doing today instead of AgentOS:**

| What they're doing | Why it breaks down |
|---|---|
| Slack channels | Agents post updates to Slack. No state transitions, no ownership, no audit trail. |
| Spreadsheets | Tracking agent configs in Google Sheets. No versioning, no sync, no visibility. |
| Custom internal dashboards | Each company with 5+ agents has built something in-house. Duplicated effort. |
| Nothing | Agents run without monitoring. Failures are discovered by downstream damage. |
| AgentOps / LangSmith | Observability tools — they can tell you what an agent did *after* the fact. No control. |
| Claude Code / Codex CLI | Session-based, solo developer tools. No team features, no persistence, no marketplace. |

**What existing frameworks are missing:**

- **Persistence**: Claude Code and Codex both start fresh every session. Agents that remember your project, team conventions, and accumulated context don't exist in any mainstream tool.
- **Team management**: No framework has solved multi-agent coordination visibility for non-technical stakeholders. The CFO cannot see what the 12 agents did this week without asking an engineer.
- **Agent interoperability**: A CrewAI agent cannot call an AutoGen agent. An n8n workflow cannot trigger a Dify agent. MCP handles tool discovery but not stateful agent communication.
- **Marketplace / exchange**: Skills marketplaces (Skillstore.io, Majestic, SkillsForge) exist. Agent (persistent, stateful, deployable) marketplaces do not.
- **Cost attribution**: Running 40 agents and knowing what each one costs, per team, per project, per cycle — unsolved by every current tool.
- **Compliance infrastructure**: SOC2-ready exports, immutable audit logs, compliance attestation. Enterprises need this before they can deploy agents broadly.

## Constraints

- **v1 validated the wedge but not the full vision**: The Agent Creation Studio (config IDE) proves the category and builds the initial user base. It's the right entry point. But it is NOT the business — it's the wedge into a much larger platform opportunity.
- **Must be agent-native**: Models how agents actually work — tool-using, stateful, probabilistic. Not a human PM tool adapted for agents.
- **Must integrate with existing agent runtimes**: Cannot reinvent the agent execution layer. MCP is the integration target; export to Claude Code YAML, OpenAI JSON, CrewAI, AutoGen must be supported.
- **Must be built for team buyers first, solo users second**: The product decisions for team buyers (RBAC, audit trails, cost attribution, cloud sync) are not add-ons — they are the core product for the primary buyer.
- **Discovery must be the moat**: A config editor is copyable in weeks. A curated, growing, community-validated database of tool-to-capability mappings that reveals emergent capabilities takes years to replicate. Everything else is commoditizable; this is the durable advantage.
- **Agents are a fleet, not a team**: Agents are infrastructure — like servers, not employees. The mental model is resource management and cost attribution, not team coordination. Agents don't need "sprints" or "morale." They need monitoring, cost attribution, and reliable execution. This shapes every UI decision: dashboards show resource utilization, not team happiness.

## Revised Premises

1. **The buyer is a business, not a hobbyist.** Team leads, startups, and enterprises are the real buyers. Indie developers are the long tail, not the core market.
2. **The category is "agent infrastructure," not "agent IDE."** An IDE is a solo developer tool. AgentOS is infrastructure for organizations running agents at scale. The v1 config studio is the entry wedge; the platform is the destination.
3. **Agent persistence across sessions is the missing primitive.** Both dominant coding agents (Claude Code, Codex) are session-based. Persistent agents that accumulate knowledge over time are the obvious next step and nobody has shipped it properly yet.
4. **The marketplace for agents doesn't exist yet — but skills marketplaces prove the model.** AI Skillstore.io (4,577+ skills) proves developers will package and share agent configurations. The leap from skills to deployable agents is the next structural step, requiring verification infrastructure first.
5. **Compliance and cost attribution are the enterprise unlock.** Enterprises will buy agent tooling when it can satisfy compliance requirements and show cost-per-agent. Without these, agent deployment stays siloed in engineering.
6. **MCP is the integration layer, not a feature.** MCP compatibility is how AgentOS connects to the runtime ecosystem. The strategic goal is: build the config layer, let MCP be the runtime bridge.
7. **"LinkedIn for agents" is 2028+.** The full marketplace/exchange vision requires: trust infrastructure, standardized capability definitions, runtime portability, legal/liability frameworks, and agent identity systems. None exist yet. Build toward it, don't claim it yet.

## Approaches Considered

### Approach A: Pure Config IDE (v1 thesis)
A focused web app for creating and managing one agent at a time. Export to YAML/JSON. Single-user, local-first.
```
Effort: M (1 dev, 4-6 weeks) | Risk: Low
Pros: Ship fast, validate the creation thesis, works with any agent framework
Cons: Doesn't solve the team management problem, doesn't address enterprise buyers
Value: Wedge only, not a platform
```

### Approach B: Full Agent Operations Platform
Build everything: config IDE + multi-agent canvas + runtime telemetry + compliance + marketplace.
```
Effort: XL (3+ devs, 12+ months) | Risk: High
Pros: Complete solution, hardest to replicate
Cons: 12+ months to anything impressive, enterprise sales cycle is long, complex to maintain
Value: Platform, but by the time you ship, competitors have caught up
```

### Approach C: Platform Wedge (Recommended)
Start with the config IDE + team features + Discovery Panel (v1). Use MCP export as the runtime bridge. Layer team collaboration, cost attribution, and compliance in v1.1-v1.2. Build toward the capability registry and marketplace incrementally. The platform emerges from validated wedges rather than being shipped all at once.
```
Effort: S (v1) → L (v1.1) → L (v1.2) → XL (v2) | Risk: Medium
Pros: Validates the wedge fast, builds the moat (Discovery) incrementally, stays competitive
Cons: Requires clear product sequencing, risk of platform scope creep without discipline
Value: Wedge → Platform; each phase funds the next
```

> Phase 1.1 (team + RBAC + cloud sync + MCP) is substantially larger than v1. Cloud sync requires conflict resolution and a backend. RBAC requires permission modeling. MCP export requires protocol R&D. Treat v1.1 as a separate release with its own validation milestones — not a sprint added to v1.

**The competitive gap this exploits:** OpenClaw tried to be everything from day one and became a "chimera mess." The right move is opinionated wedges that compound into a platform.

## Recommended Approach

**AgentOS: The Agent Operating System**

Build in phases, each validating the next:

### Phase 1 (v1): The Wedge
Agent Creation Studio + Discovery Panel + Claude Code / OpenAI export. Ship fast. Validate that teams will pay for better agent configuration UX. Local-first browser app. Solo first, team features minimal.

**This is the validation phase.** If teams don't adopt the config studio, nothing else matters. If they do, the team features are the upsell.

**The Discovery Panel UX:** When a user selects tools in the config panel, the Discovery engine (a pure function, no state) queries the capability DB for all mappings where `toolIds ⊆ selectedTools`. Results appear as cards in the Discovery Panel, sorted by emergent-first — capabilities requiring 2+ tools surface at top with a highlighted "Emergent" badge. For each capability:
- **Ready** (all required tools selected): green border, "Ready to use" label
- **Partial** (some tools selected): shows "Needs: [missing tool]" in clickable text — clicking auto-adds the missing tool to the agent config
- **Unverified** (new AI-generated entries without human validation): gray badge, "AI-generated" label

The capability DB is seeded by an AI Generator-Evaluator pipeline (see Phase 3 sketch). In v1 it ships as a curated static JSON file (~50-100 hand-curated entries). The discovery engine updates via `useMemo` reacting to tool selection state — no polling, no timers, no background fetching. Zero-latency updates.

### Phase 2 (v1.1): Team Foundation
Add team workspaces, cloud sync (Excalidraw-style), RBAC, agent sharing within teams, MCP export. Flip from solo-first to team-first. This is where the primary buyer (team lead / startup) becomes the focus.

**Effort note: L → XL.** Team cloud sync, RBAC, and MCP export together are a larger engineering scope than Phase 1. This should be treated as a distinct release with its own validation milestones, not a quick feature addition.

**Key feature: MCP export.** Once AgentOS can export MCP format, an AgentOS-configured agent runs inside ACC or any MCP-compatible runtime. AgentOS + ACC are complementary, not competing.

### Phase 3 (v1.2): Capability Registry + Verification
Public, versioned capability registry seeded by AI Generator-Evaluator pipeline. Community contributions with AI-assisted validation. Capability Verification harness. This is the moat — it compounds with every agent created and every contribution.

**This is what makes AgentOS defensible.** A config editor is copyable. A curated, growing, community-validated capability database with trust signals is not.

**The AI Generator-Evaluator pipeline (how the moat gets built):**
- **Generator Agent:** Given a set of tool definitions (name, description, category, input schema), generates candidate capability mappings — e.g., "Web Search + Page Fetch → Multi-source synthesis: cross-reference claims across multiple sources before presenting." Each candidate includes: capability name, description explaining the emergent property, the specific toolIds required, whether it's emergent (requires 2+ tools), category, and coverage_level (core/extended/experimental).
- **Evaluator Agent:** Scores each generated candidate 0-10 across 5 dimensions: outcome clarity (does it state the goal?), tool justification (why these specific tools?), when-to-prefer (when choose this over alternatives?), emergent logic (for emergent capabilities — does it articulate the emergent property?), no hallucination (are the toolIds real and correctly named?). Candidates scoring ≥8/10 are accepted; 6-7/10 are revised; ≤5/10 are rejected.
- **Pipeline:** 2 generator batches × 2 evaluator passes + human spot-check (top 20 by score + random 10 for quality gate). Output is a versioned JSON data file committed to the AgentOS repo and shipped with each release. Community contributions follow the same evaluation path via GitHub PRs with automated AI pre-screening.
- **Why this works:** The Generator-Evaluator harness (per Anthropic's published design) produces high-quality entries at machine scale with human spot-check as the final trust gate. The capability DB is seeded once by the founding team, then grows via community contributions that are AI-screened before human review.

### Phase 4 (v2): Platform
Multi-agent workspace, cost attribution per agent/team/project, compliance attestation (SOC2-ready exports), immutable audit logs, enterprise SLA on the capability registry, team scenario libraries.

**The marketplace requires all of Phase 1-3 first.** You cannot have a functional agent marketplace without: (1) verified capability definitions, (2) runtime compatibility (MCP), (3) cost attribution, (4) trust infrastructure. Build these first.

## The Platform Opportunity

### What "Platform" Means Here

AgentOS is not an app. It is a **platform layer** between agent builders and agent operators. Three distinct layers:

```
┌──────────────────────────────────────────────────────────────┐
│                    AGENT MARKETPLACE LAYER                    │
│  Browse agents by capability • Deploy to runtime • Monitor    │
├──────────────────────────────────────────────────────────────┤
│                   VERIFICATION / REGISTRY LAYER               │
│  Capability Registry • Verification harness • Trust scores  │
├──────────────────────────────────────────────────────────────┤
│                      CONFIGURATION LAYER                      │
│  Agent Studio • Discovery Panel • Team collaboration         │
├──────────────────────────────────────────────────────────────┤
│                       INTEGRATION LAYER                       │
│  MCP Export • Claude Code YAML • OpenAI JSON • CrewAI      │
└──────────────────────────────────────────────────────────────┘
```

### The Marketplace is 2028+. Here's What to Build Toward.

**Why 2028:**
The prerequisites for a true agent marketplace don't exist yet:
1. **Agent Protocol adoption** — agents must communicate with each other and marketplace infrastructure before they can be deployed by third parties
2. **Standardized capability benchmarks** — "good at research" is not a spec. Without measurable, comparable capabilities, discovery is meaningless
3. **Runtime containerization** — agents must be deployable in standardized environments before they can be listed and deployed by a third party
4. **Trust and liability framework** — legal norms for what happens when a deployed agent makes a harmful decision
5. **Agent identity system** — persistent identifiers for agents across platforms and deployments

**What is viable NOW (2026):**
- **Skills/templates marketplace**: Proven by Skillstore.io (4,577+ skills). This is configuration packages, not autonomous agents. Viable today.
- **Agent benchmarking platform**: A standardized evaluation framework for agent capabilities. Needed before any higher-level marketplace. Chargeable as a service to agent developers.
- **Agent deployment infrastructure**: "Heroku for agents" — hosting, scaling, monitoring for running agents. Infrastructure is the boring but real money.

**The bridge strategy:**
```
2026: Agent Studio (config IDE) + MCP export
     → Teams can build in AgentOS, deploy anywhere via MCP

2026-2027: Capability Registry + Verification
     → Trust infrastructure for agent capabilities
     → Teams can verify agent claims before deploying

2027-2028: Agent benchmarking + deployment infrastructure
     → Standardized evaluation framework
     → Agents can be deployed to AgentOS infrastructure

2028+: True agent marketplace
     → If all above are solid: browse, compare, contract, deploy
```

## Open Questions

### Must Resolve Before Full Platform Scoping

1. **Buyer confirmation**: Are team leads and startups actually the primary buyer, or does the product need to pivot to serve enterprises from day one? The distinction changes the entire v1.1 priority stack.

2. **Revenue model** ⚠️ BLOCKING: Local-first v1 is free. Team features in v1.1 require a business model. Subscription per team? Per agent? Per API call to cloud features? Usage-based? **This decision is blocking** — it determines whether v1.1 is a paid product launch or a freemium expansion, which changes the entire GTM, success criteria, and team hiring plan. Resolve before scoping v1.1.

3. **MCP as the runtime bridge**: Is MCP adoption going to be widespread enough by 2026-2027 to bet the integration strategy on it? Or should AgentOS build its own lightweight agent runtime?

4. **Market timing**: The prerequisites for a true agent marketplace (benchmarks, trust infrastructure, containerization) don't exist. What's the right milestone to trigger marketplace investment — a specific registry size? A specific customer count? A specific MCP adoption threshold?

5. **Competitive window**: OpenClaw is wounded. CrewAI, Dify, n8n are orchestration-focused, not management-focused. How long is the window before one of them adds team management features?

### Deferred to Implementation

- Exact pricing tiers and GTM motion
- Specific CI/CD integration points (GitHub Actions, etc.)
- Specific compliance framework coverage (SOC2, HIPAA, etc.)
- Enterprise procurement requirements

## Appendix: What the ACC Article Validated

ACC (Agent Command Center) is a real-time runtime dashboard for trading agents. 5 views (Grid, Pipeline, Focus, Activity Feed, Workspace Graph). Observer + Manager intelligence layer. MCP integration. Learning store.

**ACC validates:**
- The category is real: Karpathy publicly stated the need
- The problem is urgent: agents exist, infrastructure doesn't
- The operations layer is valuable: real-time monitoring, approval panels, learning stores

**ACC does NOT solve (and AgentOS should):**
- Config/creation layer: ACC is runtime only, not build-time
- Cross-domain portability: ACC is trading-specific
- Discovery: ACC assumes you already know what tools the agent has
- Team management: ACC is a solo operations panel, not a team collaboration tool
- Marketplace: ACC has no discovery or distribution layer

**The strategic conclusion:** AgentOS + ACC are complementary. Use AgentOS to build and configure. Use ACC (or any MCP runtime) to operate. The integration point is MCP export. The products do not compete; they complete each other.

## Appendix: What "Linear for Agents" Gets Right and Gets Wrong

**Gets RIGHT:**
- The organizational layer is missing: the thing that makes agents legible to non-engineers in the organization
- Opinionated simplicity beats configurable chaos: build what agents need, not what humans needed
- The "Jira replacement" moment: when the CFO asks "what did the agents do this week?" there should be an answer

**Gets WRONG:**
- Agents aren't humans: sprints, velocity, morale, career growth — none of this applies
- Agents don't block the way humans do: an agent waiting on input just waits; it doesn't need motivation
- The "team" framing may be wrong: are agents a team, or a single distributed system with parallel execution paths?
- Gamification doesn't apply: Linear's achievement systems don't make sense for agents

**Better framing:** "The operating system for AI agents at scale." This captures:
- Infrastructure layer (like an OS)
- Organization-wide deployment
- Management layer (OS manages resources; AgentOS manages agents)
- Leaves room for multiple product shapes: observability-first, workflow-first, compliance-first
