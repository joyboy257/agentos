## tl;dr

**AgentOS is "VS Code for agents."** Built from scratch for what AI agents actually are: living, reasoning processes — not API pipelines, not code functions. Three pillars: (1) visual canvas for composing agents, (2) integrated terminal that's the actual runtime, (3) Orchestrator agent as a first-class watchdog. MCP-native throughout.

**Runtime: Electron.** MCP via STDIO works natively in Electron's main process. `child_process.spawn()` or `UtilityProcess` API hosts agent processes. OpenSumi (`opensumi/core`, 3.6k stars) proves the stack works — AI Native IDE framework with MCP + xterm.js + React. No cloud infra, no WASM, no compromise.

---

## Problem Statement

The IDE and terminal were built for humans writing code. AI agents are not humans writing code — they are autonomous processes that *use* tools, *collaborate* with other agents, and *reason* about what to do next.

Every existing tool either:
- **Bolts AI onto a workflow canvas** (n8n, Dify, LangFlow) — workflows are static API pipelines, not agents-as-processes
- **Gives you raw code with no visual debugging** (CrewAI, AutoGen) — agents are Python classes, not first-class visual objects
- **Is terminal-only and solo** (Claude Code) — best agent experience, but no canvas, no orchestration, session-based

**Agents are none of these things.** An agent is: a persistent process, a reasoning entity, a tool user, a collaborator, a memory-bearing entity. The IDE/terminal were designed for *humans writing code*. That's the gap.

## Status Quo

Indie devs managing multi-agent setups use **tmux grids + custom scripts**: multiple terminal panes, each running an agent process, watched manually. When one fails, they see the scroll and intervene.

The workaround is:
- **Fragile** — tmux crash = visibility loss
- **Manual** — human monitoring required
- **Non-visual** — no state overview, agents are terminal output
- **Isolated** — agents don't know about each other, no handoff management

**Primary buyer:** Indie dev / solo builder (3-5 personal agents)
**Day 1 wedge:** Visual agent canvas — see all agents on a canvas with live state, connect them, see capability edges emerge
**Secondary:** Team leads, startup founders, enterprise AI engineers

Why indie dev first:
- Fastest to adopt (no procurement)
- Highest tolerance for rough edges
- Strongest evangelists if it works

## Constraints

1. **Runtime environment — RESOLVED: Electron.** Electron main process hosts MCP via `child_process.spawn()` with `stdio: ['pipe', 'pipe', 'pipe']`. `UtilityProcess` API is an alternative. OpenSumi proves this stack works in production. No cloud infra needed. No WASM needed.

2. **Orchestrator UX is unsolved:** Beginners don't know what policies to set. Solution: sensible defaults + "watch and learn" mode.

3. **Full reimagination is XL scope:** 18-24 engineer-months across three phases.

4. **No demand validation:** First users will be pioneers, not mainstream.

5. **Electron distribution:** Users install an app. No browser-first UX. Mitigation: Electron is a well-understood distribution model for developer tools.

## Glossary

| Term | Definition |
|------|------------|
| **Canvas** | Infinite workspace where agents are displayed as living nodes with real-time state |
| **Agent node** | A rectangular card on the canvas representing one agent; shows: state (idle/running/error), tool list, expandable workbench |
| **Terminal tab** | A terminal emulator tab (xterm.js) attached to an agent node; shows live stdout/stderr and reasoning trace |
| **Workbench** | The expanded view of an agent — shows reasoning trace, tool call history, current step |
| **Orchestrator** | A special-purpose agent that watches all other agents on the canvas, enforces policies, surfaces alerts |
| **Capability edge** | A labeled directed edge on the canvas between two agent/tool nodes representing a known capability (e.g., "Research Pipeline: web_search + browser → multi_source_verification") |
| **MCP server** | A process that implements the MCP protocol, exposing tools to agents (e.g., filesystem, github, memory) |
| **MCP client** | The per-agent runtime component that connects to MCP servers and calls tools |
| **Handoff** | When Agent A completes work and passes context to Agent B; shown as an animated edge on the canvas |
| **Agent process** | The underlying OS process (Node.js worker or Python subprocess) that runs an agent's reasoning loop |
| **Policy** | A rule configured on the Orchestrator: on_event → action (e.g., on_timeout → retry, max_retries=3) |

## Data Model

### Core Entities

```typescript
interface Agent {
  id: string;
  name: string;
  role: string;                    // "Research Agent", "Writer Agent"
  status: 'idle' | 'running' | 'waiting' | 'error' | 'completed';
  tools: string[];                  // MCP tool IDs assigned to this agent
  mcpServerIds: string[];          // which MCP servers this agent connects to
  workbench: WorkbenchState;        // current reasoning trace, tool call history
  memory: MemoryState;              // per-agent memory (isolated)
  createdAt: number;
  updatedAt: number;
}

interface Canvas {
  id: string;
  agents: string[];                // agent IDs in order of creation
  edges: Edge[];                   // handoff edges, capability edges
  orchestratorConfig: OrchestratorConfig;
}

interface Edge {
  id: string;
  type: 'handoff' | 'capability' | 'tool_use';
  sourceId: string;                // agent or tool ID
  targetId: string;
  label?: string;                   // for capability edges: "Research Pipeline"
  status: 'active' | 'inactive';
}

interface OrchestratorConfig {
  enabled: boolean;
  policies: Policy[];
  watchAgentIds: 'all' | string[]; // which agents to watch
}

interface Policy {
  id: string;
  trigger: 'on_timeout' | 'on_error' | 'on_long_running' | 'on_handoff_delay';
  action: 'retry' | 'skip' | 'escalate' | 'alert' | 'require_approval';
  threshold?: number;               // seconds for timeout/delay thresholds
  maxRetries?: number;
  backoff?: 'linear' | 'exponential';
  notifyChannels?: string[];        // slack, email, etc.
}

interface MCPConnection {
  id: string;
  serverType: string;               // 'filesystem', 'github', 'memory', etc.
  status: 'connected' | 'connecting' | 'error' | 'disconnected';
  tools: string[];                  // tool IDs exposed by this server
  config: Record<string, unknown>; // server-specific config
}
```

## Technical Architecture

**Stack:** Electron + React 19 + TypeScript + React Flow + xterm.js + Zustand + SQLite (better-sqlite3) + MCP SDK

**Electron is the app shell.** Main process is the process supervisor. Renderer processes are sandboxed UI only.

```
┌──────────────────────────────────────────────────────────────┐
│                    ELECTRON MAIN PROCESS                     │
│                                                              │
│  PROCESS SUPERVISOR:                                         │
│  • Spawns agent processes via child_process.spawn()         │
│  • Registry: agent ID → PID, status, MCP connections       │
│  • IPC via Electron IPC (contextBridge → preload → renderer  │
│  • Persistence: better-sqlite3 (SQLite)                    │
│  • Watchdog: restart crashed agents, notify orchestrator    │
│                                                              │
│  AGENT PROCESS (per agent):                                 │
│  • Node.js child process (or UtilityProcess)               │
│  • Connects to MCP servers via stdio                       │
│  • Reasoning loop streams output via IPC → xterm.js         │
│  • MCP servers: filesystem, github, memory, etc.            │
├──────────────────────────────────────────────────────────────┤
│                    SANDBOXED RENDERER PROCESS               │
│                                                              │
│  CANVAS LAYER: React Flow — infinite canvas, agent nodes    │
│  TERMINAL LAYER: xterm.js — receives stream from agent IPC  │
│  ORCHESTRATOR PANEL: alert feed, policy status, actions   │
│  DISCOVERY PANEL: capability cards, MCP server palette       │
└──────────────────────────────────────────────────────────────┘
```

**Security:** `contextIsolation: true`, `nodeIntegration: false`, sandboxed renderers. All privileged ops (process spawn, file access, MCP stdio) live in main process. Renderer talks to main via `ipcRenderer.invoke()` / `ipcMain.handle()`. No WebSocket server needed — Electron IPC is synchronous and fast.

## Approaches Considered

### Approach A: Canvas-First MVP
**Summary:** Visual canvas with agent nodes + terminal tabs (display only, not runtime). No orchestrator. MCP palette (config only, not execute).
- **Effort:** M | **Risk:** Low — validates canvas UX fast
- **Pros:** Proves the monitoring thesis; 8-12 weeks to v0.1
- **Cons:** Doesn't run agents; users still use tmux for runtime
- **Reuses:** React Flow, xterm.js

### Approach B: Full Reimagination — ELECTRON CHOSEN
**Summary:** Canvas + Terminal + Orchestrator + MCP, all in one product. Agents run *here*, not in a separate terminal.
- **Effort:** XL | **Risk:** Medium — scope risk remains, runtime is resolved (Electron)
- **Pros:** Categorical claim: "this IS the IDE for agents"; all three pillars ship together
- **Cons:** 18-24 weeks to v1.0
- **Stack:** Electron + React Flow + xterm.js + Zustand + SQLite + MCP SDK
- **Reference:** OpenSumi (`opensumi/core`) — AI Native IDE framework proving this stack works

### Approach C: Terminal-First with Canvas Overlay
**Summary:** Claude Code-style CLI + React Flow canvas as a toggle. Agents run in the CLI; canvas is a visualization layer.
- **Effort:** L | **Risk:** Low — builds on proven paradigm
- **Pros:** Fastest path to real; validates runtime first; power users get CLI, visual users get canvas
- **Cons:** "Flipped" positioning is harder to market; canvas is secondary
- **Reuses:** Claude Code CLI pattern, xterm.js, React Flow overlay

## Recommended Approach

**APPROACH B: Full Reimagination on Electron**

The vision is the product. The runtime is resolved: Electron main process hosts MCP via `child_process.spawn()` — proven, works, no cloud needed. The reimagination ships in three coherent phases, each with a clear exit criterion.

### Orchestrator Defaults — Watch and Learn Mode

The orchestrator ships with sensible defaults (on_timeout → retry 3x, on_error → alert). But beginners don't know what policies to configure. The orchestrator therefore has a "watch and learn" mode:

- **Observation period:** When enabled, the orchestrator logs all handoff events, tool call durations, error rates, and delay patterns for 24-72 hours
- **Policy suggestion:** After observation, the orchestrator surfaces a natural-language suggestion: "I noticed Web Search often times out after 30s. Would you like me to auto-retry on timeouts?" with a one-click accept or dismiss
- **UI:** Suggestions appear as dismissible cards in the orchestrator panel header — non-blocking, informative

This means the orchestrator is useful from Day 1 even without manual configuration.

### Phase 1: v0.1 — Prove the Canvas (8-12 weeks)
- React Flow canvas with agent nodes
- Agent nodes show: name, role, tool list, status indicator
- Basic edges: handoff connections (drag to connect)
- xterm.js terminal tabs per agent (display stdout — agents still run in user's terminal)
- MCP server palette: add/remove MCP server connections
- No orchestrator yet
- **Exit criterion:** 5 indie devs say they'd replace their tmux grid with this canvas

### Phase 2: v0.2 — Add the Runtime (8-10 weeks)
- Electron main process: spawns agent processes via `child_process.spawn()`
- Agent processes connect to MCP servers via stdio
- Live reasoning trace (stream from agent process → IPC → xterm.js)
- Process persistence: better-sqlite3 saves agent state, survives tab close
- **Exit criterion:** Agents can be created, run, and observed entirely in AgentOS (no separate terminal required)

### Phase 3: v1.0 — Add the Orchestrator (6-8 weeks)
- Orchestrator agent: watches all canvas agents, evaluates policy triggers
- Default policies: on_timeout → retry (3x), on_error → alert
- "Watch and learn" mode: orchestrator observes for 24h, suggests policies
- Human-in-the-loop: approval gate for critical actions
- Capability edges: auto-draw when agent toolset matches known capability
- **Exit criterion:** At least one orchestrator policy fires in production and user acts on it

## Open Questions

1. **Orchestrator defaults:** How does a beginner configure policies without knowing their workflow? Solution: sensible defaults (retry on timeout, alert on error) + "watch and learn" mode that logs events and suggests policies after observation.

2. **Memory model:** Per-agent isolated memory (v1) vs. canvas-level shared knowledge graph (v1.1). Decision: per-agent isolated for now. Per-agent memory is sufficient for v0.1-v1.0 because: (a) the primary pain is visibility/orchestration, not shared memory; (b) shared knowledge graph introduces complexity around conflict resolution and access control. The shared knowledge graph unlocks "crew memory" use cases in v1.1 — agents that share context across a handoff.

3. **Canvas at scale:** Fleet overview mode needed for 50+ agents — agents collapse into groups with drill-down. Implementation: React Flow's group feature + custom collapse widget.

4. **Capability edge definition:** Directed edge representing data+control flow from agent A's output to agent B's input. Not "API call" — "handoff context."

5. **Electron distribution model:** Package via electron-builder. macOS (.dmg), Linux (.AppImage, .deb), Windows (.exe). Auto-update via electron-updater. Ships with bundled Node.js runtime for MCP server compatibility.

6. **Future: Claude Code plugin integration.** oh-my-claudecode (`github.com/Yeachan-Heo/oh-my-claudecode`) proves the "Oh My X" plugin model works for Claude Code — 32 agents, 31 lifecycle hooks, skills system, MCP support. Future integration: an AgentOS plugin for Claude Code that spawns the AgentOS canvas as an external tool, giving Claude Code users a visual orchestration layer without leaving their terminal workflow.

## Success Criteria

| Phase | Criterion | Measurement |
|-------|-----------|--------------|
| v0.1 | Canvas adoption | 5 indie devs use canvas as primary monitoring interface for 1 week |
| v0.1 | Canvas engagement | Each session: ≥3 agents on canvas, ≥1 handoff edge drawn |
| v0.2 | Runtime adoption | 0 separate terminal sessions for monitored agents |
| v0.2 | Process persistence | Agent survives tab close, resumes on reopen |
| v1.0 | Policy fires | ≥1 orchestrator policy triggers per week per active user |
| v1.0 | User acts on alert | ≥50% of triggered policies → user takes action (approve/retry/skip) |

**Kill condition (v0.1):** If indie devs don't place ≥3 agent nodes on the canvas and draw ≥1 handoff edge within their first 30-minute session (tracked via analytics), the Day 1 wedge is wrong. Revisit canvas UX before proceeding to v0.2.
