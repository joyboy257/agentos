## Blocker 1 (⚠️ HIGH) — Revenue Model

**Decision required before:** Any v1.1 scoping, hiring plan, or GTM strategy

v1 is local-first and free — validated. v1.1 introduces team features that require a business model. Three options on the table, none resolved:

| Model | How it works | Risk |
|---|---|---|
| **Per-team subscription** | $X/month per team workspace | Team leads may balk if they have multiple teams |
| **Per-seat** | $X/seat/month (team members) | Indie builders have low WTP; enterprise buyers want flat-rate |
| **Usage-based** | Pay per export, per MCP connection, per API call | Complex to meter; may discourage heavy use |

**Why this blocks everything:** The revenue model determines pricing, GTM motion, and what counts as a "win." Without it, success criteria are undefined and hiring plan is undatable.

**Resolve via:** Talk to 5 team leads. Ask "what would you pay per month for team workspaces + MCP export?" Get a number, not a range.

## Blocker 2 (⚠️ HIGH) — Problem/Mode Mismatch

**The thesis says "managing agents is the problem." v1 ships a creation studio.**

This is the most structurally dangerous gap. For the primary buyer (team lead with 20 agents in production), the acute pain is:
- "I have 12 agents and I don't know what they're doing"
- "Agents fail silently and I find out from downstream damage"
- "I spent my weekend babysitting agents"

v1's creation studio solves: "I want to configure my next agent better."

These are different jobs-to-be-done for the same person at different moments in the agent lifecycle. If the team lead feels the *management* pain first and comes to AgentOS, they churn before the team features (v1.1) ship. If they feel the *building* pain first, v1 captures them and earns the relationship.

**Resolve via:** 5 team lead interviews (see interview script below). If 3 of 5 describe active management pain as their primary concern, v1.1 fleet features need to ship before the team features are marketed to team leads. If 3 of 5 describe building/configuring as their primary pain, the current sequencing holds.

## Blocker 3 (⚠️ MEDIUM) — Discovery Panel Engagement is Load-Bearing

**The entire platform thesis depends on the Discovery Panel being useful.**

If users don't engage with capability cards — don't click "Needs: X" to auto-add tools, don't discover emergent capabilities — then:
- The capability registry never grows from community contributions
- The Generator-Evaluator pipeline has no flywheel to compound
- The platform vision (Registry → Verification → Marketplace) has no foundation

**Kill condition:** 90 days post-v1.1, if Discovery Panel engagement is <20% of team sessions — pivot to Config IDE SaaS ($20-50/seat/month), abandon the platform roadmap, and stop investing in the Registry/Verification/Marketplace path.

**Measure from day 1:**
- % of sessions where a user clicks through to a Discovery Panel capability card
- % of configured agents where the user acted on a "Partial" capability (added a missing tool)
- Registry growth: community contributions per 30-day window

## Scope Decision — v1 is Final

**v1 ships:**
- Config Studio (agent configuration UI)
- Discovery Panel (capability DB with Ready/Partial/Unverified states)
- Claude Code YAML export only

**v1 does NOT ship:**
- OpenAI JSON export (v1.1)
- MCP export (v1.1 — protocol R&D scope)
- History/audit trails (v1.1 — team feature)
- Team workspaces / cloud sync (v1.1)
- RBAC (v1.1)
- Capability registry with community contributions (v1.2)

**v1.1 scope (pending revenue model decision):**
- Team workspaces with cloud sync
- MCP export
- OpenAI JSON export
- History/audit trails (basic)

**v1.2 scope:**
- Capability Registry (public, community-contributed)
- AI Generator-Evaluator pipeline for entry quality
- Capability Verification harness

**v2 scope:**
- Cost attribution per agent/team/project
- Compliance attestation (SOC2-ready exports)
- Enterprise RBAC, audit logs, immutable trails

## 5-User Interview Script

### Screening filter (all 3 required):
1. Running 3+ AI agents in production (not demo, not dev)
2. Has spent 2+ hours in the past month managing/debugging agents
3. Technically capable (reads config files, understands API keys)

### Q1: "Walk me through your last agent failure. What happened, how did you find out, how long did it take to fix?"
- **Buyer confirmed:** Describes on-call pain, silent failures, babysitting
- **Red flag:** Has automated monitoring already, or doesn't know because they're not the one managing

### Q2: "Show me how you configure a new agent today."
- **Buyer confirmed:** Uses YAML/spreadsheets/Slack; they're doing it themselves
- **Red flag:** Has a dedicated infra team; developers configure independently

### Q3: "There's a tool with a web UI that lets you configure agents and shows you what emergent capabilities your tool selection enables. Would you use it, or would your developers?"
- **Buyer confirmed:** "I'd use it and hand off configs to my team"
- **Red flag:** "My developers won't touch a web UI, they prefer code"

### Q4: "What would you pay per month to stop the pain you just described?" [Get a specific number]
- **WTP signal:** $100-2K/month for team leads; $20-100/month for indie builders
- **Red flag:** "I wouldn't pay for that" or "we'd build it ourselves"

### Q5: "This tool requires configuring agents using its export format. Your team would configure in the web UI and export to your runtime. Would your developers adopt that workflow?"
- **Buyer confirmed:** Developers adopt if UX is better than YAML
- **Red flag:** "They'd never give up their own config workflow" — means migration strategy required

## MCP Commoditization Risk (Ongoing)

**The irony:** The more successful AgentOS is at making MCP universal, the more it commoditizes its own primary value proposition.

If Anthropic ships native MCP in Claude Code and OpenAI adds MCP to ChatGPT agent builder, AgentOS's Config IDE + MCP export advantage disappears. The Discovery Panel and Capability Registry are the remaining moat — but they take years to compound.

**Watch:** If Claude Code ships with native MCP export, reassess whether AgentOS's export integrations have a differentiated half-life long enough to build the Registry moat before the export advantage closes.
