## Codebase Context

AgentOS MVP (Units 1-7): single-agent config studio, Discovery Panel, Claude Code YAML / OpenAI JSON export. Stack: Vite + React 19 + TypeScript + Tailwind + shadcn/ui + Zustand + Dexie.js. Local-first, browser-only.

ACC (Agent Command Center): real-time runtime dashboard for trading agents. 5 views (Grid, Pipeline, Focus, Activity Feed, Workspace Graph). Observer + Manager intelligence layer. MCP integration with 16 fixed trading tools. Learning store with confidence scoring. Polling every 10-45s.

**Key finding from ACC analysis:** ACC validates AgentOS's category but operates in a different lifecycle layer (runtime/ops vs build-time/discoverability). The moat is NOT the config editor — it is the Discovery Engine. A config editor is copyable in weeks; a curated, growing, community-validated capability database takes years to replicate.

**Critical open question (unresolved):** Who is the buyer — indie builder (2-5 personal agents, low urgency, low pay) or team lead/devops (agent pipelines in production, high urgency, will pay)? ACC explicitly targets the latter. This decision shapes every post-MVP priority.

## Revised Rankings (post re-brainstorm with ACC context)

### Tier 1 — Core Moat Builders

These ideas compound the Discovery Engine moat and are defensible against ACC.

---

### 1. Capability Registry **[KEEP — highest priority]**
**Description:** A public, versioned registry mapping tool IDs → capability chains. The Discovery Panel queries this registry to surface emergent capabilities. Community can contribute via PRs with AI-assisted validation. Each capability entry includes: `confidence: number`, `sourceAttribution: string`, `applicationCount: number`, `lastValidated: Date`. High-use, high-confidence capabilities surface first.
**Rationale:** The moat. Every agent created using the registry adds signal (application count, confidence votes). ACC cannot replicate this without rebuilding our entire discovery layer — they are runtime-observant, not build-time-discoverable. Network effects compound over time.
**Moat compounding:** 9/10 | **ACC defense:** 8/10 | **Depth:** 10/10
**Kill condition:** Community contribution model collapses — registry becomes stale, nobody maintains it. Governance design is a v1 prerequisite, not an afterthought.
**v2 trajectory:** Foundation for capability certification, paid discovery layers, enterprise capability policies. Teams adopt for standardization; indie builders contribute out of necessity. Naturally splits buyer question: free community registry → indie, private forks with SLA → teams.
**Status:** Unexplored — needs governance design before implementation

---

### 2. Capability Verification **[KEEP — trust as the moat]**
**Description:** A testing harness that validates capability claims — given tool X and tool Y, does the agent actually produce the claimed emergent outcome? Runs synthetic test scenarios against stubbed tool responses. Emits a confidence score. Failed verifications surface as "unverified" badges in Discovery. Verified capabilities show "Verified by N users" / "AI-generated, unverified" in the CapabilityCard footer.
**Rationale:** The trust layer on top of the registry. What separates a curated registry from a wiki. ACC integration becomes possible: verified capabilities can be monitored at runtime with confidence thresholds. The verification harness becomes the foundation for AgentOS as a capability certification body — the moat is not the IDE, it's the trust stamp.
**Moat compounding:** 8/10 | **ACC defense:** 9/10 | **Depth:** 9/10
**Kill condition:** False confidence — passing tests for capabilities that fail in production. "Verified" badge becomes synonymous with "lies." Trust collapse.
**v2 trajectory:** Third-party tool developers submit tools for official verification before surfacing in Discovery. "Verification score" becomes the trust signal across the ecosystem — like npm download counts but for capability quality. ACC integration: verified agents run with confidence monitoring. AgentOS becomes the certification body for agent capabilities, not just the IDE.
**Prerequisite:** Ship AFTER Discovery Panel has >20 verified capabilities to display. Verification without a populated Discovery Panel is a solution in search of a problem.
**Status:** Unexplored — needs test harness design

### Tier 2 — Differentiating Enhancements

These widen the moat but are not the primary defensible asset.

---

### 3. Capability Chain Visualizer **[REFRAME of Live Export Preview]**
**Description:** Live Export Preview was killed (makes config editor the center instead of Discovery; tells no differentiated story vs ACC; ACC could copy it in a weekend). Replaced with: a live visualization showing how selected tools compose into capability chains, updating as the config changes. NOT raw YAML — capability flow diagrams and tool dependency graphs. Emergent capabilities (≥2 tools) cluster at the center. Ghost nodes show tools not yet selected that would complete partial chains.
**Rationale:** Makes Discovery the center of gravity. Visualizes the "Needs: X" mechanic spatially rather than as text labels. The capability graph is AgentOS's distinctive take on export — not what the YAML looks like, but how the tools you're choosing actually compose.
**Moat compounding:** 5/10 | **ACC defense:** 6/10 | **Depth:** 5/10
**Kill condition:** Visualizes what users could discover anyway from the flat capability list. Adds UI complexity without new information.
**v2 trajectory:** Evolves into collaborative capability graph editing — two users can see the same capability map and collaboratively select tools. Teams align on capability taxonomy before configuring agents.
**Status:** Exploratory — prototype needed to validate information value

---

### 4. Taxonomy-First Authoring **[REFRAME of Capability-First Creation]**
**Description:** Capability-First Creation was killed (NL inference on an empty capability map produces nonsense recommendations; users abandon Discovery permanently). Replaced with: build Discovery as a structured taxonomy editor FIRST, then layer intent inference on top. The Discovery Panel becomes a browsable, searchable taxonomy tree where users navigate capability categories before selecting tools. "I want to research" → browse "Information Gathering" → see "Multi-source Synthesis" → select the required tools.
**Rationale:** The moat isn't the NL inference — it's the taxonomy. Build the map before the GPS. A structured taxonomy that accurately maps tools to capabilities is independently valuable and compounds with every other feature. NL interface is a v2 layer on top, not the v1 entry point.
**Moat compounding:** 7/10 | **ACC defense:** 8/10 | **Depth:** 8/10
**Kill condition:** Taxonomy is too shallow to be useful (fewer than 3 levels deep) or too broad (every capability in one bucket). Users default back to tool selection because browsing taxonomy is slower.
**v2 trajectory:** If taxonomy is solid: agents can be composed visually from verified capability chains. Discovery becomes a "capability IDE" — browse, test, and chain capabilities before touching config. Community-curated agent templates emerge from taxonomy + tool combinations. Network effect on the taxonomy IS the moat.
**Status:** Exploratory — depends on Registry existing first

---

### 5. Live Playground **[KEEP — build-time advantage]**
**Description:** A REPL-like environment within AgentOS for testing agent configs without deploying. Input a scenario, pick stubbed tools, see the reasoning chain and tool call sequence. Constrained simulation — NOT real execution, NOT a runtime. Happy path, error path, edge case scenarios.
**Rationale:** Operates at build-time where AgentOS lives and ACC cannot go. Closes the loop between Discovery (what capabilities does this tool set enable?) and testing (does this actually work?). Serves indie builder (speed to confidence) differently than team lead (scenario libraries for rigor).
**Moat compounding:** 6/10 | **ACC defense:** 7/10 | **Depth:** 9/10
**Kill condition:** Stubbed tool ecosystem cannot keep pace with real tool complexity. Playground tests pass but real configs fail in production. False confidence destroys trust.
**v2 trajectory:** Evolves into collaborative agent testing — shared scenario libraries (team leads curate, indie builders consume). Discovered agents can be playground-tested inline before export. Successful test runs feed back into Discovery Engine rankings — the flywheel closes. ACC cannot replicate this because they are runtime, not build-time.
**Status:** Unexplored — stub ecosystem design needed

### Tier 3 — Contextual Opportunities

These are right for specific buyers but don't compound the moat directly.

---

### 6. Agent Security Layer **[REFRAME of Credential Vault]**
**Description:** Credential Vault was killed (users bypass vault and embed secrets because it's faster; ACC could ship secrets management in a sprint). Replaced with: not storage, but build-time security policy enforcement. Flags hardcoded credentials in discovered configs. Policy enforcement at export (never export raw API keys). Compliance attestation on exported agents. "Security score" per agent in Discovery Panel.
**Rationale:** Operates at discovery/config layer — ACC cannot intercept build-time policy violations because they are runtime. Serves team lead/devops buyer explicitly (compliance attestation, audit trails). Indie's vault is free; enterprise pays for compliance layer.
**Moat compounding:** 5/10 | **ACC defense:** 7/10 | **Buyer clarity:** 8/10
**Kill condition:** Key management UX is so painful users disable the feature entirely.
**v2 trajectory:** Compliance certification for agent configs (SOC2-ready exports, policy gates, audit reports). Enterprise buyers (team leads) pay for attestation. Indie builders get vault for free.
**Status:** Exploratory

---

### 7. Migration Assistant **[REFRAME of Import Anything]**
**Description:** Import Anything was killed ("import theatre" — impressive demos, useless outputs, users bounce). Replaced with: ONE high-fidelity path — paste a well-structured system prompt text, have AI parse it into structured config fields (role, system prompt, suggested tools). The import is scaffolding, not a magic import. Onboarding ramp for indie builders who have working prompts they want to "agentify."
**Rationale:** The blank page is the real activation barrier. One path (paste → parse → discover) is achievable. It compounds Discovery because you're parsing AND recommending capabilities. Kills the GitHub/Notion/URL multi-path that fragments the feature.
**Moat compounding:** 3/10 | **ACC defense:** 4/10 | **Buyer clarity:** 6/10
**Kill condition:** Parsing quality is too low (users spend more time correcting than writing).
**v2 trajectory:** Community-shared migration recipes for popular agent frameworks. "Config diff" — see what changed when you imported vs the default.
**Status:** Exploratory

## Revised Cross-Cutting Syntheses

### The Compounding Stack
**Registry → Verification → Taxonomy**
Registry seeds the map. Verification stamps the trust. Taxonomy makes it navigable. Together they are: the map (Registry), the trust layer (Verification), the interface (Taxonomy). This is AgentOS's durable moat — not any single feature, but the stack that no competitor can replicate without rebuilding all three layers simultaneously.

### The Flywheel
**Discovery → Playground → Registry**
Discovered capabilities can be playground-tested inline before export. Successful test runs feed back into Discovery rankings (via application count + confidence). More usage → better rankings → more discovery. This is the flywheel that turns AgentOS from a tool into a platform.

### The Buyer Split
- **Indie Builder:** Free tier: Taxonomy browsing + Migration Assistant + Live Playground + community Registry
- **Team Lead/DevOps:** Paid tier: Agent Security Layer (compliance attestation) + private Registry forks (SLA) + team scenario libraries
- The split emerges organically from features, not from a product decision.

---

## Strategic Recommendations

1. **Ship Registry before Live Export Preview.** The moat compounds from the registry outward, not from the editor inward. Every week spent on Live Export Preview (raw YAML) is a week not spent on the only thing that widens the moat.

2. **Resolve the buyer question before scoping further.** If the buyer is indie builder → prioritize Migration Assistant + Taxonomy + Playground. If team lead → prioritize Security Layer + Registry governance + team features. Building for both splits focus and weakens both.

3. **Verification after Registry, not before.** Verification without a populated Discovery Panel is a solution in search of a problem. Seed with 20+ capabilities from the AI Generator-Evaluator pipeline (Unit 5B) before shipping verification.

4. **Kill Live Export Preview as YAML preview.** The pivot to Capability Chain Visualizer is the right direction — but validate with a prototype before committing. If the graph doesn't convey information the flat list doesn't already convey, kill it and ship the list.

5. **The MCP export (v1.1) is the bridge to ACC.** Once AgentOS can export MCP format, an AgentOS-configured agent runs inside ACC. The products are complementary, not competing. Invest in MCP export as the integration point, not as a competitive threat.
